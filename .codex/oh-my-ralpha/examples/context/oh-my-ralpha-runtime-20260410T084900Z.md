# oh-my-ralpha Runtime Context Snapshot

- Task statement: Upgrade the standalone `oh-my-ralpha` repository from a packaged skill bundle into a self-contained JS runtime that carries the minimum OMX capabilities it depends on.
- Desired outcome: A new Codex environment can install and use `oh-my-ralpha` with built-in install/doctor/init/state/trace/route/planning scaffolds, while preserving Ralph-derived workflow semantics and executable guardrails.
- Known facts/evidence:
  - The standalone repo already contains the packaged skill, flow doc, example `.codex/oh-my-ralpha` truth-source files, and Node contract tests.
  - The current installer writes a launcher that points back to the source repository checkout instead of an installed runtime payload.
  - The current route gate lets vague prompts bypass planning when any planning artifacts exist.
  - The current `initWorkspace()` return contract can report a context path that was not created.
  - Original `oh-my-codex` exposes persistent state and tool surfaces through native hooks plus MCP server registration via setup.
- Constraints:
  - Implement using JS inside the standalone `oh-my-ralpha` repo.
  - Keep diffs reviewable and test-backed.
  - Do not require the full `oh-my-codex` runtime just to use the standalone repo.
  - Preserve Ralph-derived behavior rather than replacing it with a lighter workflow.
- Unknowns/open questions:
  - How much of original OMX setup/hook/MCP behavior should be mirrored in the standalone repo now versus staged later.
  - Which minimal MCP/hook surfaces are necessary for a “new Codex directly usable” bar.
