# oh-my-ralpha @LOG Hook Context Snapshot

- Task statement: Add a session-level `@LOG` debugging hook to `oh-my-ralpha` so that when the user types `@LOG`, the current Codex session begins recording hook/turn activity for later debugging.
- Desired outcome: A fresh or active Codex session can enable logging with `@LOG`, after which native hooks, notify payloads, and tool-use surfaces write a durable session log under `.codex/oh-my-ralpha/working-model/logs/` so behavior shifts after skill usage can be diagnosed.
- Known facts/evidence:
  - Current standalone runtime already owns `SessionStart`, `UserPromptSubmit`, and `Stop` native hooks.
  - Codex native hooks also expose `PreToolUse` and `PostToolUse`.
  - Codex can fire a top-level `notify` command after each agent turn; original OMX uses that path to capture `input-messages`, `last-assistant-message`, `turn-id`, and related metadata.
  - Current standalone setup does not write any `notify` config and does not register `PreToolUse` / `PostToolUse`.
  - Current standalone package already has state, trace, native hook, setup, and MCP infrastructure that can host the logging feature.
- Constraints:
  - Keep the feature session-scoped by default.
  - Do not overwrite user-owned config/hook content.
  - Prefer explicit `@LOG` activation rather than always-on logging.
  - Preserve current release-quality test coverage.
- Unknowns/open questions:
  - Whether to support an explicit disable token like `@UNLOG` in the first version.
  - Whether the release preflight should require notify logging to be enabled by default or only when configured.
- Likely codebase touchpoints:
  - `src/native-hook.mjs`

  - `test/setup-integration.test.mjs`
